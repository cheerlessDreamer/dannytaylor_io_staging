Kaisa — Audience Builder

Company: Kaisa (formerly Freespee) · Role: Lead Product Designer · Timeline: June 2022–present

The Problem

Kaisa's product team needed a segmentation engine — an "Audience Builder" — to let CRMs create targeted customer groups. The PM had spoken with internal users and written user stories that seemed straightforward. When the requirements reached me, I immediately saw problems: several features were technically impossible within the agreed timeline, and future products on the roadmap depended on this shipping on schedule. The board wasn't willing to be roadblocked for months.

My Approach

I went directly to the engineers to understand the real technical limits. Then back to the PM to separate genuine requirements from nice-to-haves. Once I had a clear picture of both sides, I designed the complete "ideal" interface — the version with everything — and worked backwards from there.

I created multiple scope packages, stripping away features to find combinations that still felt like a real, usable product rather than a hollow v1. For each version, I mapped the pre-requisites needed to unlock the next stage — things like rebuilding how the database was queried to support features like visualising audience changes over time.

Before presenting anything to my boss or the PM, I shared the proposals with the developers first. We're a small company and I can't afford to lose their trust — I need to make sure nothing I design is a surprise to them later when they're told to build it.

The Solution

A v1 Audience Builder that deliberately doesn't do everything — but is designed so that adding things later is easy.

The interface is intentionally sparse. The core of it is a form that supports nested criteria groups, even though we currently have no reason to nest anything. That structure is there for when we do. Similarly, every input that presents a list of options uses a combobox — a dropdown with a search field at the top — rather than a plain dropdown. A plain dropdown works fine with ten options; it breaks down at thirty. Designing for the current list would have meant a rework the moment the list grew, so I skipped that version entirely.

These decisions compound: because the components already accommodate more complexity, the next engineer to add a feature probably won't need to touch the design at all. That matters more than it sounds. A v1 that requires a designer to sign off on every increment tends to stay at v1 forever — the coordination cost is too high. This one is more likely to actually evolve.

Impact

The first version was released and received well by internal users. The phased roadmap I created is still actively driving development priorities — future improvements come directly from that planning document.

What I Learned

Designing the ideal version first and working backwards is more effective than building up from the minimum. You end up with a better-shaped v1 because you can see what's being deferred and why, rather than guessing what might be needed later. Also: pre-validating with engineers before presenting to stakeholders avoids surprises and builds the kind of trust that makes everything move faster.
